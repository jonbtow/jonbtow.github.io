Programming Style Guidelines

(Excerpt From
"The Elements of Programming Style" by Kernighan and Plauger)

  1. Write clearly - don't be too clever.
  1. Say what you mean, simply and directly.
  2. Use library functions whenever feasible.
  3. Avoid too many temporary variables.
  4. Write clearly - don't sacrifice clarity for efficiency.
  5. Let the machine do the dirty work.
  6. Replace repetitive expressions by calls to common functions.
  7. Parenthesize to avoid ambiguity.
  8. Choose variable names that won't be confused.
  9. Avoid unnecessary branches.
  10. If a logical expression is hard to understand, try transforming it.
  1. Choose a data representation that makes the program simple.
  2. Write first in easy-to-understand pseudo language; then translate into whatever language you have to use.
  3. Modularize. Use procedures and functions.
  4. Avoid goto's completely if you can keep the program readable.
  5. Don't patch bad code - rewrite it.
  6. Write and test a big program in small pieces.
  1. Use recursive procedures for recursively-de ned data structures.
  2. Test input for plausibility and validity.
  3. Make sure input doesn't violate the limits of the program.
  4. Terminate input by end-of-file marker, not by count.
  5. Identify bad input; recover if possible.
  6. Make input easy to prepare and output self-explanatory.
  7. Use uniform input formats.
  8. Make input easy to proofread.
  9. Use self-identifying input. Allow defaults. Echo both on output.
  1. Make sure all variable are initialized before use.
  2. Don't stop at one bug.
  3. Use debugging compilers.
  4. watch out for off-by-one errors.
  5. Take care to branch the right way on equality.
  6. Be careful if a loop exits to the same place from the middle and the bottom.
  7. Make sure your code does "nothing" gracefully.
  8. Test programs at their boundary values.
  9. Check some answers by hand.
  10. 10.0 times 0.1 is hardly ever 1.0.
  11. 7/8 is zero while 7.0/8.0 is not zero.
  12. Don't compare floating point numbers solely for equality.
  1. Make it right before you make it faster.
  2. Make it fail-safe before you make it faster.
  3. Make it clear before you make it faster.
  4. Don't sacrifice clarity for small gains in "efficiency".
  5. Let your compiler do the simple optimizations.
  6. Don't strain to re-use code; reorganize instead.
  7. Make sure special cases are truly special.
  8. Keep it simple to make it faster.
  9. Don't diddle code to make it faster - and a better algorithm.
  10. Instrument your programs. Measure before making "efficiency" changes.
  1. Make sure comments and code agree.
  2. Don't just echo the code with comments - make every comment count.
  3. Don't comment bad code - rewrite it.
  4. Use variable names that mean something.
  5. Use statement labels that mean something.
  6. Format a program to help the reader understand it.
  7. Document your data layouts.
  8. Don't over-comment.
